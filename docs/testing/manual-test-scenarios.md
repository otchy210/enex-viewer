# 手動テストシナリオ（T-014）

## 1. 目的

MVP の主要ユーザーフローを人間の実操作で確認し、受け入れ可否を判定する。

## 2. 実施前提

- `nvm use` 済み
- `npm install` 済み
- `npm run dev` で API/Web が起動済み
- API: `http://localhost:3001`
- Web: `http://localhost:5173`

## 3. テストデータ

- `small.enex`: ノート約 10 件
- `medium.enex`: ノート約 100 件
- `broken.enex`: XML 破損データ

## 4. シナリオ

### MT-001 正常アップロード

- 手順:
  1. Web を開く
  2. `small.enex` を選択してアップロード
- 期待結果:
  - 処理が完了し、エラー表示が出ない
  - 一覧表示に必要な情報（importId/件数）が確認できる

### MT-002 一覧表示

- 手順:
  1. `small.enex` 取り込み後の一覧を確認
- 期待結果:
  - タイトル、日時、タグ/抜粋が表示される
  - 件数表示が取り込み結果と整合する

### MT-003 検索

- 手順:
  1. 一覧画面で既知キーワードを入力して検索
  2. 非一致キーワードでも検索
- 期待結果:
  - 一致ノートのみ表示される
  - 非一致では 0 件表示となる

### MT-004 ノート詳細

- 手順:
  1. 一覧から任意ノートを選択
- 期待結果:
  - タイトル、日時、タグ、本文、添付情報が表示される
  - 表示崩れや危険な HTML 実行がない

### MT-005 解析エラー（壊れた ENEX）

- 手順:
  1. `broken.enex` をアップロード
- 期待結果:
  - エラーメッセージが表示される
  - 画面が操作不能にならない

### MT-006 体感性能（中規模 ENEX）

- 手順:
  1. `medium.enex` をアップロード
  2. 一覧表示、検索、詳細表示を操作
- 期待結果:
  - 著しい遅延やフリーズがない

### MT-007 レスポンシブ表示

- 手順:
  1. 開発者ツールでモバイル幅（例: 390x844）へ切替
  2. アップロード、一覧、詳細を確認
- 期待結果:
  - 主要操作が可能
  - 文字切れ/重なりが致命的でない
